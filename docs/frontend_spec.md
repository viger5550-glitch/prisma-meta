# Frontend Spec: Prisma

## 1) Scope и цель

Этот документ фиксирует минимальную frontend-спеку для Prisma:
- единая дизайн-система;
- переиспользуемые UI-компоненты;
- структура мета-страницы;
- структура продуктовых лендингов;
- MVP-реализация CTA и оплаты через ЮKassa redirect.

Принцип: одна система для мета-страницы и всех лендингов, без визуальной перегрузки.

## 2) Дизайн-система (токены + принципы)

## 2.1 Цветовые токены

Базовая палитра: черный, белый, серый. Без ярких акцентов.

```json
{
  "color.bg.base": "#FFFFFF",
  "color.bg.surface": "#F6F6F6",
  "color.bg.inverse": "#111111",
  "color.text.primary": "#111111",
  "color.text.secondary": "#5C5C5C",
  "color.text.inverse": "#FFFFFF",
  "color.border.default": "#D9D9D9",
  "color.border.strong": "#B3B3B3",
  "color.action.primary.bg": "#111111",
  "color.action.primary.text": "#FFFFFF",
  "color.action.secondary.bg": "#FFFFFF",
  "color.action.secondary.text": "#111111",
  "color.action.secondary.border": "#111111",
  "color.state.hover.overlay": "rgba(0,0,0,0.04)",
  "color.state.disabled.bg": "#EDEDED",
  "color.state.disabled.text": "#9A9A9A",
  "color.focus.ring": "#111111"
}
```

Правила:
- фон страницы: `color.bg.base`;
- карточки/секции: `color.bg.surface` или `color.bg.base` с границей;
- основной текст: `color.text.primary`;
- вторичный текст: `color.text.secondary`;
- все интерактивные элементы используют единые action-токены.

## 2.2 Типографика

Шрифты:
- `font.family.base`: `Inter, Arial, sans-serif`;
- `font.family.heading`: `Inter, Arial, sans-serif` (тот же, чтобы сохранить минимализм).

Токены размеров и межстрочных интервалов:

```json
{
  "font.size.h1": "48px",
  "font.line.h1": "56px",
  "font.size.h2": "36px",
  "font.line.h2": "44px",
  "font.size.h3": "28px",
  "font.line.h3": "36px",
  "font.size.h4": "22px",
  "font.line.h4": "30px",
  "font.size.body.lg": "18px",
  "font.line.body.lg": "28px",
  "font.size.body.md": "16px",
  "font.line.body.md": "24px",
  "font.size.body.sm": "14px",
  "font.line.body.sm": "20px",
  "font.weight.regular": 400,
  "font.weight.medium": 500,
  "font.weight.semibold": 600
}
```

Иерархия:
- `h1`: название страницы/продукта;
- `h2`: ключевые секции;
- `h3`: внутренние подблоки;
- `h4`: заголовки карточек/малых блоков.

Правило для UI-лейблов (обновление):
- кнопки (`primary`/`secondary`) и короткие label в карточках используют более тонкий вес `font-weight: 500` (`font-medium`);
- `font-semibold` и `font-bold` для CTA-кнопок не используются.

## 2.3 Отступы и сетка

Базовый шаг: `8px`.

Токены spacing:

```json
{
  "space.1": "8px",
  "space.2": "16px",
  "space.3": "24px",
  "space.4": "32px",
  "space.5": "40px",
  "space.6": "48px",
  "space.8": "64px",
  "space.10": "80px",
  "space.12": "96px"
}
```

Правила вертикального ритма:
- расстояние между крупными секциями: `space.8`-`space.12`;
- расстояние между заголовком и текстом секции: `space.2`-`space.3`;
- расстояние между карточками: `space.2`-`space.3`.

Контейнеры:
- `container.max`: `1200px`;
- горизонтальные поля: `24px` desktop, `16px` mobile;
- контент лендинга: 1 колонка с ограничением по ширине текста (до `760px` для длинных описаний).

## 2.4 Принципы UI

- минимализм: только функциональные элементы;
- карточная модель: продукты и смысловые блоки представлены карточками/секциями;
- фокус на тексте: ясная иерархия, короткие формулировки;
- отсутствие визуального шума: минимум декоративной графики и эффектов.

## 3) Core UI-компоненты

Компоненты должны быть едиными для мета-страницы и продуктовых лендингов.

## 3.1 `Card` (карточка продукта)

- Назначение:
  - показать продукт в каталоге;
  - дать быстрый переход на лендинг.

- Структура:
  - `title`;
  - `description`;
  - `price` (опционально);
  - `link`/CTA.

- Props:
  - `title: string` (required);
  - `description: string` (required);
  - `price?: string`;
  - `link: string` (required);
  - `onClick?: () => void`;
  - `ariaLabel?: string`.

- Поведение:
  - вся карточка кликабельна;
  - click/enter -> переход на `link`;
  - `hover`: легкое усиление тени/границы;
  - `active`: короткое уменьшение масштаба/контраста;
  - фокус-стейт обязателен для клавиатурной навигации.

## 3.2 `Button`

- Назначение:
  - унифицированная точка действия (CTA).

- Варианты:
  - `primary` (основной сценарий: оплата/вход в продукт);
  - `secondary` (альтернативный сценарий: подробнее/назад/другой путь).

- Props:
  - `variant: "primary" | "secondary"` (required);
  - `size?: "md" | "lg"` (default `md`);
  - `disabled?: boolean`;
  - `href?: string`;
  - `onClick?: () => void`;
  - `type?: "button" | "submit"`.

- Поведение:
  - `hover`: изменение фона/границы по токенам;
  - `disabled`: не кликается, muted-стили;
  - если передан `href`, работает как ссылка с button-стилем.
  - вес шрифта для текста кнопки: `font-medium` (единый стандарт для всего сайта).

## 3.3 `Header`

- Назначение:
  - глобальная навигация верхнего уровня.

- Структура:
  - логотип `Prisma` (слева);
  - опциональная навигация (справа).

- Props:
  - `showNav?: boolean`;
  - `navItems?: Array<{ label: string; href: string }>`;
  - `sticky?: boolean`.

- Поведение:
  - клик по логотипу всегда ведет на `/`;
  - навигация скрывается или упрощается на mobile.

## 3.4 `Section`

- Назначение:
  - контейнер смыслового блока лендинга/мета-страницы.

- Структура:
  - optional `eyebrow`;
  - `title`;
  - `content`;
  - optional `actions`.

- Props:
  - `id?: string`;
  - `title?: string`;
  - `description?: string`;
  - `children: ReactNode`;
  - `tone?: "default" | "surface"`.

- Поведение:
  - единые вертикальные отступы;
  - единые max-width и внутренние paddings.

## 3.5 `DetailModal` (модалка подробностей)

- Назначение:
  - универсальный паттерн раскрытия деталей карточки внутри Prisma без перехода на отдельную страницу;
  - используется на странице челленджа (`/challenge`) и должен переиспользоваться для всех остальных карточек, у которых есть подробное описание.

- Когда использовать:
  - карточка имеет разворачиваемый контент (что входит, как устроено, ритуалы, формат) и не требует отдельного URL/лендинга;
  - быстрый просмотр деталей без потери контекста радиальной композиции;
  - не использовать для оплаты, аутентификации, длинных лонгридов и шагов с состоянием.

- Когда НЕ использовать:
  - если для продукта уже есть полноценный лендинг — модалка не должна заменять лендинг;
  - если контент требует SEO/прямой ссылки — нужна отдельная страница.

- Структура (сверху вниз):
  - `title` — название модуля/карточки в serif;
  - `eyebrow` / `subtitle` — подзаголовок карточки, uppercase, tracking;
  - `divider` — короткая линия (~96px) под подзаголовком;
  - `body` — последовательность блоков двух типов:
    - `paragraph` — обычный текст;
    - `list` — список с тире (`—`) в начале каждого пункта;
  - `closeButton` — иконка-крестик в правом верхнем углу.

- Размеры и поведение:
  - максимальная ширина: `460px`;
  - максимальная высота: `88vh` со скроллом внутри;
  - фон страницы под модалкой: `rgba(0,0,0,0.30)`;
  - паддинги контента: `32-40px`;
  - типографика: serif для `title`, Inter для остального; subtitle `10px / tracking 0.32em`, body `14px / line-height 1.7`.
  - вертикальный ритм шапки модалки фиксированный: отступ `title -> subtitle` должен быть равен отступу `subtitle -> divider` (сейчас `mt-7` и `mt-7`);
  - у `title` не добавляется дополнительный верхний отступ внутри контентного контейнера.

- Дублирование карточки:
  - `title` в модалке должен точно повторять `titlePrimary` карточки;
  - `eyebrow` / `subtitle` в модалке должен точно повторять `titleSecondary` карточки;
  - порядок в модалке всегда: заголовок карточки, затем подзаголовок карточки, затем divider и подробный текст.

- Триггер открытия:
  - кнопка `MORE` на карточке;
  - если у карточки есть `detail`, кнопка ведет себя как `<button>` и открывает модалку;
  - если у карточки нет `detail`, кнопка остается ссылкой (`<Link>`) на внешний/внутренний URL.

- Поведение модалки:
  - закрытие по `Esc`;
  - закрытие по клику на затемненный фон;
  - закрытие по клику на крестик;
  - блокировка прокрутки `body` пока модалка открыта;
  - не закрывается при клике внутри контента;
  - `role="dialog"`, `aria-modal="true"`, `aria-label = title`.

- Props (контракт компонента):
  - `detail: { eyebrow: string; title: string; blocks: Block[] } | null`;
  - `onClose: () => void`;
  - где `Block = { type: "paragraph"; text: string } | { type: "list"; items: string[] }`.

- Правила консистентности:
  - все карточки челленджа, у которых есть подробное описание, должны открывать его именно через `DetailModal`, а не через отдельную страницу или собственную верстку;
  - если в будущем понадобится оверлей/шторка для других сущностей (продукт, FAQ, секция знаний), используется тот же визуальный паттерн (eyebrow / serif title / divider / blocks) и то же поведение (Esc/backdrop/scroll-lock);
  - один глобальный паттерн модалки = одно поведение во всем сайте; новые модалки не создаются ad hoc.

## 3.6 `Layout`

- Назначение:
  - общий каркас страницы.

- Структура:
  - `Header`;
  - `main` (контент);
  - optional `footer`.

- Props:
  - `children: ReactNode`;
  - `withFooter?: boolean`;
  - `container?: "default" | "wide"`.

- Поведение:
  - обеспечивает консистентные контейнеры;
  - задает общий фон и вертикальный ритм.

## 4) Мета-страница Prisma (UI и логика)

## 4.1 Структура страницы

- `Header`: логотип и опциональная навигация;
- `Hero`: название Prisma + краткое описание;
- `Products Orbit Block`: композиция карточек продуктов.

## 4.2 Логика карточек

- каждая карточка представляет отдельный продукт;
- данные карточки: `title`, `description`, `format`, `link`;
- click на карточку -> переход на соответствующий лендинг внутри Prisma.

## 4.3 Композиция

- центр: ядро системы (Prisma как хаб);
- вокруг: карточки продуктов (орбита);
- линии/связи между центром и карточками допустимы только как минимальные графические подсказки.

## 4.4 Поведение

- `hover`: подчеркивает интерактивность карточки;
- кликабельность: карточка + внутренний CTA ведут в один и тот же лендинг;
- адаптив:
  - desktop/tablet: радиальная или полурадиальная композиция;
  - mobile: вертикальный список карточек (без круговой схемы).

## 5) Лендинг продукта (структура + UI)

Лендинг работает как самостоятельная точка входа и как часть общей системы Prisma.

Обязательные блоки:
1. `Hero`:
   - название продукта;
   - короткое описание;
   - основной CTA.
2. `Для кого` (аудитория/сегмент).
3. `Что внутри` (состав/модули/формат контента).
4. `Формат / процесс` (как пользователь проходит путь).
5. `Результат` (что получает пользователь).
6. `Цена` (если применимо).
7. Финальный `CTA` (повтор основного действия).

Глобальные требования:
- логотип Prisma всегда доступен и ведет на главную;
- единый стиль с мета-страницей (токены, типографика, компоненты);
- блок связанных продуктов обязателен, чтобы страница не была тупиком.

## 6) CTA и оплата через ЮKassa (MVP)

## 6.1 Кнопка "Оплатить"

Поток:
1. пользователь нажимает `Оплатить`;
2. фронтенд вызывает endpoint создания платежа (`POST /api/payment/create`);
3. backend возвращает `paymentUrl` (redirect URL ЮKassa);
4. фронтенд выполняет redirect пользователя на ЮKassa.

MVP-правила:
- без сложных статусов и оркестрации;
- минимальная валидация входных параметров перед вызовом;
- idempotency обрабатывается на backend-стороне.

## 6.2 После оплаты

- ЮKassa возвращает пользователя на страницу успеха Prisma (например, `/payment/success?product=...`);
- success-страница подтверждает оплату и дает следующий шаг в продукт.

## 6.3 Success page

Содержимое:
- подтверждение: "Оплата прошла успешно";
- пояснение: что произойдет дальше;
- primary CTA: переход в продукт (бот/платформа/домен);
- secondary CTA: вернуться на главную Prisma.

## 6.4 Ошибка / отмена

- при отмене/ошибке пользователь возвращается на лендинг продукта (`/product-x?payment=cancelled`);
- показывается понятное сообщение;
- доступны действия:
  - повторить оплату;
  - выбрать другой продукт;
  - вернуться на главную.

## 7) Правила навигации и связности

- логотип всегда ведет на `/`;
- карточки на мета ведут на лендинги;
- CTA с лендинга ведут во внешний продукт или в оплату;
- после внешнего шага пользователь получает путь назад в Prisma.

Базовое правило:
- ни одна страница не должна быть тупиком;
- на каждой странице минимум два осмысленных выхода:
  - целевой CTA;
  - переход в экосистему (главная или связанные продукты).

## 8) Минимальный checklist реализации

- [ ] токены цветов/типографики/spacing вынесены в единый источник;
- [ ] `Card`, `Button`, `Header`, `Section`, `Layout` переиспользуются везде;
- [ ] мета-страница имеет адаптивную орбиту (mobile -> список);
- [ ] все лендинги собраны из одинаковых структурных блоков;
- [ ] сценарий ЮKassa redirect -> success/cancel покрыт;
- [ ] нет тупиковых страниц;
- [ ] на внешних точках есть возврат в Prisma;
- [ ] все карточки с подробным описанием открывают `DetailModal`, а не самописные оверлеи.
